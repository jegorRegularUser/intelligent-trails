Введение

Современная web-разработка – это мир технически сложных, но вместе с тем удивительно простых и понятных решений. На сегодняшний день программисты создали достаточное количество прикладных средств программирования, способных не только облегчить работу разработчика, но и ускорить общее время разработки проекта. Вместе с этим в последние годы особенно заметно выделилось направление интеллектуальных систем планирования и оптимизации маршрутов: всё чаще и чаще появляются сервисы, предлагающие пользователям готовые маршруты, однако большинство из них ориентированы либо на автомобильную навигацию, либо на простое построение кратчайшего пути между двумя точками, не учитывая интересы пользователя и возможность посещения интересных мест по пути.

В данной выпускной квалификационной работе рассмотрено создание интеллектуальной системы построения пешеходных и велосипедных маршрутов с автоматическим включением точек интереса. Система позволяет пользователю указать начальную и конечную точки маршрута, выбрать категории мест, которые он хотел бы посетить (кафе, музеи, парки, достопримечательности и т.д.), после чего автоматически строит оптимальный маршрут с учётом направления движения, качества мест и разнообразия альтернатив. Система включает в себя веб-интерфейс на базе современного фреймворка Next.js 16, интеграцию с картографическими сервисами Yandex Maps и OpenStreetMap, а также систему аутентификации и персонального хранения маршрутов.

Разработанная система позволяет достигнуть следующие цели:
• экономия времени на планирование прогулок и поездок с посещением интересных мест;
• автоматический подбор оптимальных точек интереса по заданным категориям с учётом направления движения;
• возможность оценки времени и расстояния маршрута до его прохождения;
• создание единой централизованной базы данных для хранения построенных маршрутов пользователя;
• возможность делиться маршрутами через URL-ссылки;
• поддержка нескольких видов транспорта (пешком, велосипед, общественный транспорт, автомобиль);
• предоставление альтернативных вариантов для каждой точки маршрута.

Для достижения поставленных целей решены следующие задачи:
• бизнес-моделирование процессов предметной области;
• анализ требований к системе и её проектирование;
• разработка концептуальной, логической и физической моделей данных;
• реализация алгоритма интеллектуального ранжирования мест по качеству, направлению и разнообразию;
• интеграция с внешними API (Yandex Maps, OpenStreetMap Overpass API);
• разработка системы аутентификации с поддержкой OAuth-провайдеров;
• оптимизация производительности через кэширование и параллельную обработку запросов;
• покрытие кода автоматизированными тестами.

В настоящий момент существуют веб-системы с похожим функционалом, такие как Google Maps, Yandex Maps, 2GIS и другие. Однако эти системы предлагают пользователю либо построение кратчайшего маршрута между двумя точками, либо поиск отдельных мест на карте. Ни одна из них не предоставляет возможности автоматического построения маршрута с включением точек интереса по заданным категориям. Пользователю приходится вручную искать интересные места, добавлять их в маршрут и проверять, не слишком ли большой получается крюк. Это особенно неудобно при планировании длительных прогулок или велопоездок, когда хочется посетить несколько мест разных категорий (например, парк, кафе и музей), но не хочется тратить время на ручной подбор оптимального порядка посещения.

Учитывая вышесказанное, можно определить главную цель и актуальность разработки. Цель работы заключается в создании интеллектуальной системы, способной автоматически строить оптимальные маршруты с точками интереса на основе предпочтений пользователя. Цель определяется актуальностью работы: в современном мире, где люди всё больше ценят своё время и стремятся к активному образу жизни, возможность быстро спланировать интересную прогулку или поездку с посещением нужных мест становится важным преимуществом. Также разработанная система будет полезна туристам при планировании маршрутов в незнакомых городах, студентам при организации досуга, и всем, кто хочет эффективно совмещать перемещение с посещением интересных мест.


1 Анализ предметной области

Маршрут (от фр. marche route — «путь следования») — заранее намеченный или установленный путь следования, направление движения. В контексте пешеходных и велосипедных прогулок маршрут представляет собой последовательность географических точек, соединённых путями перемещения, с указанием способа передвижения, расстояния и времени в пути.

Современные картографические сервисы предоставляют пользователям возможность построения маршрутов между двумя точками с учётом различных видов транспорта. Однако процесс планирования прогулки с посещением нескольких интересных мест остаётся трудоёмким: пользователю необходимо самостоятельно найти подходящие места, оценить их расположение относительно маршрута, добавить их в качестве промежуточных точек и проверить, насколько увеличилось время и расстояние. Этот процесс может занимать от 15 до 30 минут для маршрута с 3-4 точками интереса.

В разработке современных картографических приложений принимает участие большое количество специалистов – frontend-разработчики, backend-разработчики, специалисты по работе с геоданными, дизайнеры интерфейсов. Вначале проектируется архитектура системы, выбираются картографические API и источники данных о местах. Затем разрабатывается пользовательский интерфейс, реализуется логика построения маршрутов, интегрируются внешние сервисы. После этого система тестируется на различных сценариях использования и оптимизируется для обеспечения быстрой работы.

Первые системы навигации появились в 1980-х годах и были ориентированы исключительно на автомобильную навигацию. С развитием интернета и мобильных технологий появились веб-сервисы и мобильные приложения для построения маршрутов. Однако фокус большинства из них остаётся на построении кратчайшего или быстрейшего пути, без учёта интересов пользователя.

В наше время маршрутные сервисы используются миллионами людей ежедневно. Они помогают ориентироваться в незнакомых местах, планировать поездки, оценивать время в пути. Однако для планирования досуга и прогулок с посещением интересных мест существующие решения недостаточно удобны. Пользователю приходится либо вручную искать места и добавлять их в маршрут, либо использовать отдельные сервисы для поиска мест и построения маршрутов, что создаёт дополнительные неудобства.

Рабочий процесс планирования прогулки с посещением интересных мест в существующих системах выглядит примерно следующим образом:
• определение начальной и конечной точек маршрута;
• поиск интересных мест в районе предполагаемого маршрута;
• оценка расположения найденных мест относительно прямого пути;
• добавление выбранных мест в качестве промежуточных точек;
• проверка итогового времени и расстояния;
• корректировка маршрута при необходимости.

• предоставлять альтернативные варианты для каждой точки маршрута;
• рассчитывать время и расстояние маршрута с учётом выбранного вида транспорта;
• обеспечивать возможность сохранения и совместного использования маршрутов;
• поддерживать несколько языков интерфейса (русский, английский);
• обеспечивать аутентификацию пользователей через различные провайдеры (email/пароль, Google, GitHub, Yandex).


2 Разработка технического задания

2.1 Область применения
Информационная система разработана для широкого круга пользователей и предназначена для автоматизации процесса планирования пешеходных и велосипедных маршрутов с посещением интересных мест.

2.2 Назначение и цели создания системы

2.2.1 Назначение системы
Разрабатываемая система будет состоять из общественной части (веб-интерфейса в браузере), позволяя пользователям строить интеллектуальные маршруты с точками интереса, и персональной части, которая позволит зарегистрированным пользователям сохранять построенные маршруты, просматривать историю и делиться маршрутами с другими пользователями.

2.2.2 Цели создания системы
Информационная система создаётся с целью упрощения и автоматизации процесса планирования прогулок с посещением интересных мест, а также для экономии времени пользователей при подборе оптимальных точек интереса по заданным категориям.

2.3 Требования к информационной системе

2.3.1 Исходные данные
Для создания информационной системы Intelligent Trails необходимы следующие исходные данные:
• API-ключ Yandex Maps для построения маршрутов;
• доступ к OpenStreetMap Overpass API для поиска мест;
• база данных MongoDB для хранения пользователей и маршрутов;
• OAuth-ключи провайдеров аутентификации (Google, GitHub, Yandex).

2.3.2 Функциональные требования
Система должна удовлетворять следующим требованиям:
• клиент-серверное приложение с веб-интерфейсом;
• реализация алгоритма интеллектуального подбора точек интереса;
• адаптивный поиск мест по радиусу с расширением до нахождения достаточного количества вариантов;
• ранжирование мест по качеству, направлению и расстоянию;
• веб-интерфейс для построения маршрутов с указанием начальной, конечной точек и категорий мест;
• отображение маршрута на интерактивной карте Yandex Maps;
• предоставление альтернативных вариантов для каждой точки маршрута;
• расчёт времени и расстояния маршрута с учётом выбранного вида транспорта;
• кодирование маршрута в URL для возможности совместного использования;
• система аутентификации с поддержкой email/пароль и OAuth-провайдеров;
• сохранение построенных маршрутов в базу данных;
• просмотр истории маршрутов с фильтрацией и сортировкой;
• поддержка двух языков интерфейса (русский, английский);
• ведение базы данных (ввод, вывод и редактирование данных);
• разграничение прав доступа к системе.

2.3.3 Требования к безопасности и надёжности системы
Для обеспечения безопасности и надёжности информационной системы должны быть предусмотрены:
• хеширование паролей с использованием bcrypt;
• защита от CSRF-атак через NextAuth.js;
• валидация пользовательского ввода на клиенте и сервере;
• информирование пользователя об ошибках при работе с программой;
• ограничение прав доступа пользователя к объектам системы;
• обработка ошибок внешних API с fallback-механизмами.

Система должна адекватно реагировать на неверно заданные исходные данные, поддерживать диалоговый режим в рамках предоставляемых пользователю возможностей.
Система должна корректно реагировать на удаление, изменение данных, то есть обеспечивать целостность данных. В случаях неправильных действий со стороны пользователя система должна выдавать соответствующие предупреждения.

2.4 Условия эксплуатации

2.4.1 Требования к техническому и программному обеспечению
Для корректной работы информационной системы необходимо выполнение следующих требований:
• операционная система: MS Windows 10/11, macOS 10.15+, Linux;
• процессор Intel Core i3 или аналогичный;
• оперативная память 4096 Мбайт и выше (рекомендуется 8 Гб и выше);

• среда выполнения Node.js 18.17 или выше;
• веб-браузер: Google Chrome 90+, Firefox 88+, Safari 14+, Edge 90+;
• подключение к интернету для доступа к картографическим API.

2.6 Архитектура системы

Система построена по модульной архитектуре с чётким разделением ответственности между компонентами:

Слой представления (Presentation Layer):
• React-компоненты для отображения интерфейса;
• разделение на layout-компоненты, feature-компоненты и UI-компоненты;
• использование серверных компонентов для начальной загрузки данных.

Слой бизнес-логики (Business Logic Layer):
• Server Actions для обработки пользовательских действий;
• сервисы для работы с внешними API (OSM, Yandex);
• алгоритмы ранжирования и оптимизации маршрутов.

Слой данных (Data Layer):
• репозитории для работы с MongoDB;
• схемы данных и валидация;
• индексы для оптимизации запросов.

Слой интеграции (Integration Layer):
• адаптеры для внешних API;
• система кэширования для снижения нагрузки на внешние сервисы;
• обработка ошибок и fallback-механизмы.

2.7 Алгоритм интеллектуального построения маршрутов

Ключевой особенностью системы является алгоритм интеллектуального подбора точек интереса, который включает:

1. Адаптивный поиск по радиусу:
Система последовательно расширяет радиус поиска (200м → 800м → 2500м → 6000м) до тех пор, пока не найдёт минимум 5 подходящих мест. Это обеспечивает нахождение мест даже в районах с низкой плотностью объектов.

2. Ранжирование по качеству:
Каждое место получает оценку качества на основе:
• наличия рейтинга и его значения;
• популярности (наличие Wikipedia, веб-сайта, телефона);
• размера объекта (для парков и природных зон).

3. Фильтрация по направлению:
Места, находящиеся в направлении конечной точки (в конусе ±90° от прямого пути), получают приоритет. Это предотвращает добавление мест, требующих значительного отклонения от маршрута.

4. Штраф за расстояние:
Места, находящиеся далеко от прямого пути, получают штрафные баллы:
• 0-500м: без штрафа;
• 500-1000м: -10 баллов;
• 1000-2000м: -20 баллов;
• более 2000м: -30 до -50 баллов.

5. Обеспечение разнообразия:
Из топ-20 мест по качеству выбираются 5 альтернатив с минимальным расстоянием между ними не менее 300 метров. Это даёт пользователю реальный выбор между разными вариантами.

2.8 Оптимизация производительности

Для обеспечения быстрой работы системы реализованы следующие оптимизации:

Параллельная обработка запросов:
Геокодирование нескольких адресов выполняется параллельно через Promise.all(), что ускоряет обработку в 3 раза по сравнению с последовательным выполнением.

In-memory кэширование:
Результаты запросов к Overpass API кэшируются в памяти на 10 минут. Это ускоряет повторные запросы в 15-20 раз и снижает нагрузку на внешний сервис.

Умное геокодирование:
Система использует адреса из данных OpenStreetMap, когда они доступны, что снижает количество запросов к Yandex Geocoder API на 60%.

Сжатие данных в URL:
Маршруты кодируются в URL с использованием сжатия LZ-string и однобуквенных ключей, что позволяет уменьшить длину URL в 3-4 раза.

React Compiler:
Экспериментальный компилятор React автоматически оптимизирует компоненты, добавляя мемоизацию там, где это необходимо.

2.9 Тестирование

Система покрыта автоматизированными тестами (69 тестов):
• тесты базы данных (34 теста) – проверка CRUD-операций, индексов, валидации;
• тесты Server Actions (28 тестов) – проверка бизнес-логики сохранения маршрутов, обновления профиля, аутентификации;
• тесты сервисов (7 тестов) – проверка работы с OSM API, обработки категорий, адаптивного поиска.

Для изоляции тестов используется MongoDB Memory Server, который создаёт временную in-memory базу данных для каждого тестового набора.
