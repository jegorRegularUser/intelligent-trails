2.Разработка модулей личного кабинета пользователя и истории маршрутов для информационной системы умного планировщика прогулок

2.1 Общее назначение личного кабинета

Личный кабинет является центральным элементом управления учетной записью. Он предназначен для предоставления персонализированного доступа к настройкам профиля, параметрам безопасности и системным предпочтениям. В отличие от публичной части приложения, доступ к данному модулю предоставляется исключительно авторизованным пользователям.

Визуально личный кабинет представляет собой отдельную страницу, структурированную в виде набора информационных карточек, каждая из которых отвечает за свой изолированный функционал (Рисунок 2.1).

2.2 Ролевая модель и структура данных

В проекте предусмотрена единая категория пользователей — «Пользователь». Это обусловлено спецификой сервиса, ориентированного на индивидуальное планирование маршрутов без сложной иерархии прав и громоздкого административного функционала.

Для хранения данных используется документо-ориентированная СУБД MongoDB. Благодаря использованию библиотеки Mongoose, в проекте реализована строгая типизация схемы данных на уровне приложения. Схема пользователя включает обязательные поля (имя, email), идентификатор провайдера авторизации (локальная регистрация или внешняя через Яндекс) и языковые предпочтения. Использование Mongoose позволяет автоматически валидировать данные до их сохранения в базу, предотвращая, например, запись некорректных форматов электронной почты.

Пример описания интерфейса пользователя (из файла types/user.ts):

export interface IUser {
name: string;
email: string;
provider: 'local' | 'yandex';
locale: 'ru' | 'en';
savedRoutes?: string[];
createdAt?: Date;
}

2.3 Общая структура личного кабинета

Модуль личного кабинета динамически объединяет несколько независимых функциональных блоков на одной странице. Для удобства навигации интерфейс разделен на логические секции. Основные компоненты модуля и их назначение представлены в таблице 2.1.

Таблица 2.1 — Структура функциональных блоков личного кабинета
Название компонента	Назначение и функции интерфейса
Блок «Личные данные»	Просмотр текущего email и редактирование имени пользователя через текстовые поля.
Блок «Безопасность»	Форма изменения пароля (ввод старого и нового) или вывод информационного баннера при авторизации через Яндекс.
Блок «Настройки»	Форма с настройками различных визуальных аспектов системы: языка текста сайта, отображения расстояний в км или милях и тп.

2.4 Архитектура и безопасность личного кабинета

Система безопасности построена на серверных технологиях фреймворка Next.js. Проверка авторизации и загрузка персональных данных происходят исключительно на сервере до того, как страница будет отправлена в браузер клиента. Это полностью исключает возможность несанкционированного доступа.

Ключевой код проверки авторизации (из файла app/profile/page.tsx):

export default async function ProfilePage() {
const session = await getServerSession();
if (!session) {
redirect('/login');
}
// Дальнейшая загрузка компонентов профиля
}

Особенности реализации безопасности:

    Серверная изоляция: персональные данные не передаются в браузер до успешного прохождения проверки авторизации.

    Автоматическое перенаправление: неавторизованные пользователи немедленно перенаправляются на страницу входа.

2.4.1 Обновление личных данных

Интерфейс обновления личных данных представляет собой карточку с полями ввода для имени и адреса электронной почты (Рисунок 2.2). Поле email заблокировано для редактирования в целях сохранения уникальности аккаунтов, в то время как поле имени доступно для изменения.


Рисунок 2.2 – Блок просмотра и редактирования личных данных

Для изменения имени пользователя применяется функционал серверных действий (Server Actions). По нажатию на кнопку «Сохранить изменения» форма напрямую вызывает функцию, которая взаимодействует с базой данных MongoDB.

Обработка запроса на обновление профиля (из файла actions/user.ts):

export async function updateProfileName(formData: FormData) {
const session = await getServerSession();
const newName = formData.get('name');
await User.findByIdAndUpdate(session.user.id, { name: newName });
revalidatePath('/profile');
}

После успешного обновления записи в базе данных вызывается функция обновления кэша. Благодаря этому страница не перезагружается целиком, а интерфейс пользователя мгновенно отображает новое имя.

2.6 Интеграция компонентов и итоговая сборка страницы профиля

Финальным этапом разработки личного кабинета стала сборка визуальных компонентов в единую страницу. Корневой файл выполняет роль контроллера: он обращается к базе данных, получает данные пользователя и передает их в карточки интерфейса в виде свойств.

Такая архитектура исключает появление задержек при поэтапной загрузке данных и визуального дергания элементов на экране, так как страница формируется на сервере целиком.

Фрагмент корневого компонента страницы профиля (из файла app/profile/page.tsx):

import { getTranslations } from "next-intl/server";
import { getServerSession } from "@/lib/auth";
import { ProfileSettings } from "@/components/features/ProfileSettings";
import { SecuritySettings } from "@/components/features/SecuritySettings";

export default async function ProfilePage() {
const session = await getServerSession();
const t = await getTranslations("Profile");

return (
<div className="min-h-screen bg-slate-50 py-12 px-4">
<h1 className="text-3xl font-extrabold">{t("title")}</h1>

  <div className="flex flex-col gap-6 mt-8">
    <ProfileSettings initialName={session.user.name} initialEmail={session.user.email} />
    <SecuritySettings provider={session.user.provider} />
  </div>
</div>

);
}

В данном модуле родительский файл отвечает исключительно за получение данных из базы, а карточки отвечают за отрисовку интерфейса. Это обеспечивает быструю работу личного кабинета.
